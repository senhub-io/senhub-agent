# SenHub Agent 0.4.1

:material-calendar: 2026-06-23 · auto-update reliability fix

A focused point release that makes the built-in auto-updater actually
apply. Two fixes stack up: the active periodic updater was a no-op on the
current (local) configuration and never resolved a new release, and the
install path staged the daemon binary where the hardened service could not
overwrite it. With both landed, an enabled agent downloads, verifies and
restarts onto the new binary with no manual step.

<div class="rn-filter"></div>


## Fixed

<ul class="rn">
<li><span class="tag t-fixed">Fixed</span> <span class="tag t-area">auto-update</span> The active periodic updater was a <strong>no-op</strong>: the agent's version was resolved as empty, so every tick concluded "No update required" and the registry's <code>latest</code> was never fetched — only the passive startup check queried the registry, and that only logs a warning. The updater now resolves its target to <code>latest</code> by default (a new <code>auto_update.version</code> may pin a specific target); the channel still resolves the stable list only, betas are never auto-applied. (#567)</li>
<li><span class="tag t-fixed">Fixed</span> <span class="tag t-area">install</span> <code>agent install</code> now stages the daemon binary under <code>/var/lib/senhub-agent/bin</code> instead of <code>/usr/bin</code>, so the unprivileged <code>senhub</code> service user can write its own replacement. Under the hardened unit the old location was read-only (<code>ProtectSystem=full</code>) and root-owned, and the apply failed with "read-only file system" / "permission denied". The packaged units (<code>refresh-unit</code>, <code>.deb</code>, <code>.rpm</code>) use the same path so all three agree. (#571)</li>
<li><span class="tag t-fixed">Fixed</span> <span class="tag t-area">install</span> The systemd unit's restart policy is now <code>always</code> instead of <code>on-failure</code>: the updater exits cleanly to hand off to the freshly written binary, and <code>on-failure</code> would not restart a clean exit — a self-updated agent could have stayed down. An explicit <code>systemctl stop</code> is still honored. (#567)</li>
</ul>

## Internal

<ul class="rn">
<li><span class="tag t-internal">Internal</span> <span class="tag t-area">config</span> The removed pre-0.2.0 online/offline-mode vocabulary is purged from the CLI, config, code and current docs — the <code>mode</code> field is dropped (existing keys ignored), the status "connection mode" is removed, and <code>RemoteConfigurationData</code> is renamed to <code>ConfigurationData</code>. The agent is always locally configured; no behavioral change.</li>
</ul>
