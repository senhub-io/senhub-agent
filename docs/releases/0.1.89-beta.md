# Release 0.1.89-beta

**Date:** 2026-05-06
**Type:** Beta Release (major feature)
**Branch:** feat/otlp-export → dev

## Overview

Adds **native OTLP/gRPC push export** of metrics and logs to SenHub
Agent. Operators can now ship the agent's data to any OpenTelemetry
collector, vmagent, or commercial OTLP gateway (Grafana Cloud,
Honeycomb, …) without standing up a Prometheus scrape. This is the
second OTel-first sink — Prometheus 0.1.88 shipped first as a pull
endpoint; OTLP ships now as a push exporter.

**Crucial property:** OTLP and Prometheus carry the **same metric
names, types, units, attributes**. Both consume the neutral
`OtelRecord` stream from the new `otelmapper` package. Operators
querying Prometheus today and switching to OTLP tomorrow do not
rewrite a single PromQL query.

## Highlights

- Brand-new `otlp` storage strategy with full OTel SDK pipeline
  (otlpmetricgrpc + otlploggrpc + BatchProcessor + LoggerProvider).
- Strict OTel spec compliance — gRPC transport, cumulative counter
  semantics, standard resource attributes, RFC 5424 severity mapping.
- Full TLS / mTLS / Bearer header authentication options out of the
  box, mapped 1:1 to OTel SDK options.
- Logs signal: `syslog` and `event` probes now publish OTel log
  records to a process-wide pub/sub channel that the OTLP strategy
  drains and ships.
- New **`linux_logs` probe** that reads the local systemd journal
  (`journalctl --output=json --follow`) and publishes log records to
  the same channel — Linux-only, no CGO, no extra deps.
- Refactor: extracted `Resolve()` and `OtelRecord` from the prometheus
  package into a neutral `otelmapper` package so that Prometheus and
  OTLP share the same vocabulary without sink-to-sink dependency.
- Full live validation against a real otel-collector ingesting into
  VictoriaMetrics + VictoriaLogs (Phase 4).

## Major Changes

### OTLP storage strategy

**New package:**
`internal/agent/services/data_store/strategies/otlp/`

```
config.go              — typed Config + ParseConfig
client.go              — buildExporters (otlpmetricgrpc + otlploggrpc)
                         with TLS, mTLS, headers, compression, retry
strategy.go            — OTLPSyncStrategy (SyncStrategy interface impl)
                         lifecycle: idempotent Start/Shutdown, periodic
                         metrics push, graceful drain
metric_store.go        — strategy-local LWW cache
metrics_exporter.go    — pushOnce → otelmapper.Resolve →
                         metricdata.ResourceMetrics → exporter.Export
resource.go            — service.name / service.instance.id /
                         service.version / deployment.environment
                         + operator extras
logs.go                — logsPipeline (SDK BatchProcessor +
                         LoggerProvider) + logsPump (drains
                         agentstate channel, calls logger.Emit)
```

Type → Aggregation mapping (per OTel spec):
- `counter`        → `Sum{Cumulative, Monotonic}`
- `updowncounter`  → `Sum{Cumulative, !Monotonic}`
- `gauge`          → `Gauge`

Counter `start_time_unix_nano` is the strategy's start time —
counter resets on agent restart, standard OTel semantics.

### Configuration

```yaml
storage:
  - name: otlp
    params:
      endpoint: "otel-collector.internal:4317"   # required
      headers:
        Authorization: "Bearer ${OTLP_TOKEN}"
      tls:
        enabled: true
        ca_file: /etc/ssl/private/otlp-ca.pem
        cert_file: /etc/ssl/private/agent.pem    # mTLS, optional
        key_file:  /etc/ssl/private/agent.key
      compression: gzip
      timeout: 10s
      retry:
        enabled: true
        initial_interval: 5s
        max_interval: 30s
        max_elapsed_time: 1m
      signals:
        metrics:
          enabled: true
          interval: 30s
          temporality: cumulative
        logs:
          enabled: true
          batch_size: 1000
          batch_timeout: 5s
          buffer_size: 10000
      resource:
        service.name: senhub-agent
        deployment.environment: prod
```

Adding an `otlp` storage block is purely additive. Existing PRTG /
Nagios / SenHub / HTTP storages are unaffected.

### otelmapper extraction (refactor)

**New package:**
`internal/agent/services/data_store/otelmapper/`

`OtelRecord`, `CacheMetric`, `ResolveOptions`, `CacheReader`,
`DefinitionLookup`, `Resolve()`, `convertValue()` moved out of the
prometheus package. Both Prometheus and OTLP now consume from this
neutral mapper — no sink-to-sink dependency. The Prometheus
exposition output is byte-identical post-refactor.

### Logs path (Phase 3)

**New file:** `internal/agent/services/agentstate/log_channel.go`

Process-lifetime pub/sub for log records. Producers (any probe) call
`PublishLog`; consumers `SubscribeLogs(buf)`. Drop-oldest under
backpressure with a global drop counter. Defines a
transport-neutral `LogRecord` so probes don't transitively pull in
the OTel SDK.

Includes the canonical RFC 5424 → OTel severity table:
`SyslogPriorityToSeverity` / `SyslogPriorityToText`.

### Probe wiring for logs

- **`syslog` probe**: every received syslog message becomes an OTel
  log record with severity-mapped attributes
  (`syslog.facility`, `syslog.hostname`, `syslog.appname`, …) plus
  producer identity. Unchanged: continues emitting `syslog_event`
  DataPoints to the existing event strategy.
- **`event` probe**: every accepted HTTP event becomes a log record
  with severity from the event's `severity` field (EMERG..DEBUG)
  and payload fields under the `senhub.event.*` namespace.
  Unchanged: continues emitting `event_event` DataPoints.

### New `linux_logs` probe

**New package:** `internal/agent/probes/linuxlogs/`

Reads the local systemd journal via `journalctl --output=json
--no-pager --follow` and publishes each entry as an OTel log record.

Operator knobs (all optional):
```yaml
- name: system-logs
  type: linux_logs
  params:
    units: [ssh.service, nginx.service]   # optional unit filter
    identifiers: [sshd, kernel]            # optional ident filter
    priority: 4                            # max severity (0..7)
    include_boot: false                    # default: only new records
```

Attributes mapped to OTel semantic conventions: `host.name`,
`systemd.unit`, `syslog.appname`, `process.pid`, `process.owner.uid`,
`process.executable.name`, `systemd.transport`.

Linux-only via build tags (`//go:build linux`). Stub on darwin /
windows registers but fails loudly with clear error on Start. Operators
can keep one config across mixed-OS deployments without conditional
logic.

### Probe routing

Default `GetTargetStrategies()` updated from
`[senhub, prtg, http]` → `[senhub, prtg, http, otlp]` on
BaseProbe and 12 metric probes. Strategies that aren't configured are
silently skipped by the data_store router, so this is purely additive
— agents without OTLP keep behaving exactly as before.

## Live validation (Phase 4)

End-to-end tested against sha901's otel-collector + VictoriaMetrics +
VictoriaLogs:

1. SSH forward tunnel localhost:4317 → sha901:4317.
2. Agent ran locally with cpu+memory probes and storage [http, otlp].
3. **Metrics arrived in VictoriaMetrics** with the expected names:
   `system_memory_usage_bytes`, `system_memory_limit_bytes`,
   `system_memory_utilization_ratio`, `system_linux_cpu_load_*`.
   Resource attributes (service.name, service.instance.id,
   service.version, deployment.environment) attached to every series.
   24 GB memory used reported live from the test Mac.
4. **Logs arrived in VictoriaLogs** via `cmd/otlp-validate` (a
   minimal SDK-only tool) with severity=WARN, body, custom
   attributes, stream keyed on (service.name,
   deployment.environment).

The proof point: metrics arrived in VM with the same names as the
Prometheus exposition's internal records (after the collector's
standard dot→underscore + unit-suffix conversion).

## Backward compatibility

- **PRTG / Nagios / SenHub strategies**: zero behavioral change.
- **Prometheus exposition**: byte-identical output post otelmapper
  extraction; same names, same labels, same headers.
- **JWT licenses**: every probe type ID is preserved (`logicaldisk`,
  `cpu`, `linux_logs`, …).
- **Configuration files**: enabling OTLP is purely additive
  (`storage: [..., otlp]` + `otlp:` sub-block).
