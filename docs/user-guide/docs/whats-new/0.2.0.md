# SenHub Agent 0.2.0

:material-calendar: 2026-05-19 · <span class="tag t-breaking">Breaking</span> major release

A major housekeeping release that simplifies how the agent is **configured**
and **packaged**. Nothing changes in what the agent monitors — but a few flags
and file locations moved, so **if you already run an agent, read the
[upgrade guide](#upgrade-guide) before you update.**

!!! highlight "Highlights"
    - **Simpler to run** — the agent now loads its configuration only from
      local files. The cloud-configuration path is gone, so there's one less
      thing to reason about.
    - **Cleaner install** — configuration and logs now live where each
      operating system expects them, and `install` creates the full folder
      layout for you (no more manual `mkdir`).
    - **Tidier downloads** — every release ships exactly one ZIP per platform,
      with the binary always named the same inside.

## Breaking changes

<ul class="rn">
<li><span class="tag t-breaking">Breaking</span> <span class="tag t-area">CLI</span> The <code>--offline</code>, <code>--authentication-key</code> and <code>--server-url</code> flags were <strong>removed</strong>. The agent rejects unknown flags at startup, so update your systemd <code>ExecStart</code> or Windows service path <strong>before</strong> upgrading — otherwise the service won't start. See the <a href="#upgrade-guide">upgrade guide</a>.</li>
<li><span class="tag t-breaking">Breaking</span> <span class="tag t-area">config</span> The <strong>default config path moved</strong> to an OS-canonical location. Installs that pass <code>--config-path</code> explicitly are unaffected; installs that relied on the old default (<code>&lt;binary&gt;/agent-config.yaml</code>) must move the file or keep passing the flag.</li>
<li><span class="tag t-breaking">Breaking</span> <span class="tag t-area">packaging</span> Releases now ship <strong>ZIP-only</strong>, one per platform, and the binary inside is always <code>senhub-agent</code> / <code>senhub-agent.exe</code> (no platform suffix). Tooling that downloads by URL must switch to <code>senhub-agent-&lt;os&gt;-&lt;arch&gt;.zip</code> and add an unzip step.</li>
<li><span class="tag t-breaking">Breaking</span> <span class="tag t-area">auto-update</span> A 0.2.0+ agent <strong>cannot auto-update from a pre-0.2.0 release</strong> (the old download URL no longer exists). Upgrading from 0.1.x needs <strong>one</strong> manual replace; from 0.2.0 onward auto-update resumes normally.</li>
</ul>

??? note "OS-canonical paths — reference"
    | OS | Default config | Log directory |
    |---|---|---|
    | Linux | `/etc/senhub-agent/agent.yaml` | `/var/log/senhub-agent/` |
    | Windows | `%ProgramData%\SenHub\agent.yaml` | `%ProgramData%\SenHub\logs` |
    | macOS | `/usr/local/etc/senhub-agent/agent.yaml` | `/Library/Logs/SenHub` |

## Improved

<ul class="rn">
<li><span class="tag t-improved">Improved</span> <span class="tag t-area">install</span> <code>agent install</code> now eager-creates the full layout — <code>probes.d/</code>, <code>strategies.d/</code> and the log directory — so the first run finds everything in place.</li>
<li><span class="tag t-improved">Improved</span> <span class="tag t-area">paths</span> The Linux log directory moved to <code>/var/log/senhub-agent/</code>. Existing installs keep working (it's created on first write), and uninstall now cleans both the old and new locations.</li>
</ul>

## Removed

<ul class="rn">
<li><span class="tag t-removed">Removed</span> <span class="tag t-area">config</span> The cloud/SaaS configuration path is gone — the agent is <strong>offline-only</strong> and loads its config from local YAML exclusively.</li>
</ul>

## Internal

<ul class="rn">
<li><span class="tag t-internal">Internal</span> Deleted the remote/SaaS configuration loader (641 lines) and its symbols (<code>RemoteConfiguration</code>, <code>DetectAgentMode</code>, <code>GetServerUrl()</code>, …). The cloud intake URL is now build-injected. Shared type names are kept for now; the rename is deferred to 0.2.1.</li>
<li><span class="tag t-internal">Internal</span> <span class="tag t-area">test</span> Tightened the test suite (removed mode-detection, flag-parsing and remote-config tests; added OS path helpers). <code>make test</code> green, <code>go vet</code> clean, cross-compiles on all five targets.</li>
</ul>

## Upgrade guide

!!! warning "Upgrading from 0.1.x is a one-time manual step"
    Auto-update can't bridge the pre-0.2.0 → 0.2.0 gap. Follow the runbook
    once; afterwards auto-update works normally.

1. **Stop** the `senhub-agent` service.
2. **Update the service definition** to drop the removed flags:

    ```diff title="systemd unit"
    - ExecStart=/usr/local/bin/senhub-agent run --offline --config-path /etc/senhub-agent/agent.yaml
    + ExecStart=/usr/local/bin/senhub-agent run --config-path /etc/senhub-agent/agent.yaml
    ```

    ```diff title="Windows service binPath"
    - "C:\SenHub\senhub-agent.exe" run --offline
    + "C:\SenHub\senhub-agent.exe" run
    ```

3. **Config path** — if you relied on the old default (`<binary>/agent-config.yaml`),
   move the file to the [canonical location](#breaking-changes) or keep passing
   `--config-path`.
4. **Download** `senhub-agent-<os>-<arch>.zip` from the release.
5. **Unzip** and replace the binary.
6. **Start** the service.
7. **Verify** with `senhub-agent version` and by tailing the log file.

## Known follow-ups

- The `RemoteConfigurationData` / `RemoteConfiguration*` type names will be
  renamed away from the "Remote" prefix in a 0.2.1 cleanup.
- In-place auto-upgrade from 0.1.x is intentionally not supported — use the
  one-time manual replace above.
