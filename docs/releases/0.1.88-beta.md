# Release 0.1.88-beta

**Date:** 2026-05-02
**Type:** Beta Release (major feature)
**Branch:** feat/prometheus-otel-mapping → dev

## Overview

Adds a **Prometheus / VictoriaMetrics scrape endpoint** to SenHub Agent.
Metrics are exposed in the canonical Prometheus text exposition format
(v0.0.4) on a `/metrics` route, with **OpenTelemetry-aligned naming**
across all 15 supported probes. This is the first concrete step of the
broader **OTel-first** evolution of the agent: the YAML transformer
definitions now carry an `otel:` block as the source of truth for metric
semantics, and Prometheus is the first of several mappers (PRTG, Nagios,
future Zabbix, future native OTLP push) that derive their output from it.

## Highlights

- Brand-new `/metrics` endpoint (Bearer header **or** `?token=` query
  parameter auth) alongside the existing
  `/api/{key}/prometheus/metrics` route — no breaking change to PRTG /
  Nagios.
- 15 probes mapped to OTel semantic conventions where applicable
  (`system.cpu.*`, `system.memory.*`, `system.network.*`,
  `system.filesystem.*`, `hw.*`) with documented `senhub.*` extensions
  for vendor-specific domains (NetScaler, Citrix, Veeam, Redfish).
- 8 self-observability metrics exposed by the agent itself
  (`senhub_agent_uptime_seconds`, probes count/health, collect errors,
  HTTP requests, build info, …).
- **Strict OTel emission** for status enums (drive health, NetScaler
  vServer state, Veeam job status, etc.): one data point per possible
  state with value 1 or 0 — `hw.status` pattern, ready for future OTLP
  export with zero query rewrite.
- Comprehensive end-to-end validation against a real VictoriaMetrics
  scrape: 18 metric families, 34 series ingested, PromQL `rate()` works
  on counters, label collapse via OTel attributes confirmed.

## Major Changes

### Prometheus exposition mapper

**New package:**
`internal/agent/services/data_store/strategies/http/prometheus/`

Reads the shared `MetricCache`, resolves each entry through the
extended YAML transformer definitions, and serializes Prometheus text
exposition v0.0.4. Round-trip validated on every CI build via
`expfmt.TextParser` (the canonical Prometheus parser).

**Conversion rules** follow the official OTel → Prometheus spec:
- `senhub_` prefix (frozen, non-configurable).
- Dots → underscores in names AND labels.
- Unit suffixes: `s` → `_seconds`, `By` → `_bytes`, `1` → `_ratio`,
  `bit/s` → `_bits_per_second`, `By/s` → `_bytes_per_second`.
- Counters get `_total` suffix.
- Annotated UCUM units in braces (`{packet}`, `{error}`) drop the
  brackets.
- Mapper-side value conversions: `%` → ratio (÷100), `MB` → `By`
  (×1048576), `KB` → `By` (×1024), `Mbps` → `bit/s` (×1e6),
  `ms` → `s` (÷1000), `μs` → `s` (÷1e6), `hours` → `s` (×3600).

### YAML transformer schema extended

Every probe YAML in
`internal/agent/services/data_store/transformers/definitions/` gains:

```yaml
- name: cpu_user                         # internal probe key (unchanged)
  channel: cpu_user_time                 # PRTG channel (unchanged)
  unit: "s"                              # PRTG unit (unchanged)
  description: "..."                     # display + Prometheus HELP
  otel:                                  # NEW — strict OTel target shape
    name: system.cpu.time
    unit: s
    type: counter
    attributes:
      cpu.mode: user
  tag_to_attribute:                      # NEW — tag → OTel attribute mapping
    core: cpu.logical_number
```

For status enums:

```yaml
otel:
  name: hw.status
  unit: "1"
  type: updowncounter
  attributes:
    hw.type: physical_disk
  expand:                                # strict-OTel expansion
    attribute: hw.state
    mapping:
      ok: 0
      degraded: 1
      failed: 2
      unknown: 3                          # extension value (Redfish-specific)
```

For metrics that should NOT appear in Prometheus (event conduits like
syslog/event/otel — they relay logs, they don't produce metrics):

```yaml
otel:
  skip: true
  reason: "Event conduit probe — relayed events, not metrics."
```

### Configuration

```yaml
storage:
  - name: http
    params:
      endpoints: [prtg, web, nagios, prometheus]   # add "prometheus"
      prometheus:
        include_probe_tags: true        # default — propagate custom_tags
        expose_host_metrics: true       # default — keep cpu/memory/network/filesystem
```

`expose_host_metrics: false` filters out probes flagged `host_level: true`
in their YAML (`cpu`, `memory`, `network`, `logicaldisk`) — useful when
running a node_exporter alongside on the same host.

### Routes & authentication

| Route | Auth |
|---|---|
| `GET /metrics` | `Authorization: Bearer <agent-key>` header **or** `?token=<agent-key>` query parameter (constant-time validated) |
| `GET /api/{agent-key}/prometheus/metrics` | Agent key embedded in URL (SenHub pattern) |

`401` response on missing/invalid token includes
`WWW-Authenticate: Bearer realm="senhub-agent"`.

### Agent self-observability

8 metrics emitted regardless of configured probes:

| Metric | Type |
|---|---|
| `senhub_agent_uptime_seconds` | gauge |
| `senhub_agent_cache_entries` | gauge |
| `senhub_agent_probes_active` | gauge |
| `senhub_agent_probes_total` | gauge |
| `senhub_agent_probes_healthy` | gauge |
| `senhub_agent_collect_errors_total` | counter |
| `senhub_agent_http_requests_total{endpoint=…}` | counter |
| `senhub_agent_build_info{version, commit}` | gauge=1 |

A new `internal/agent/services/agentstate/` package owns the shared
process-lifetime state (counters, active probes snapshot) without
creating an import cycle between `probes` and the http strategy.

### Robustness — never block on missing mapping

When a probe emits a metric that has no `otel:` block (rare, would
indicate a missing definition in the YAML), the agent:
- Skips the metric from `/metrics` (does not emit it).
- Logs a single `WARN` per `(probe_type, metric_name)` pair (deduplicated
  via a package-level `sync.Map`).
- Continues serving all other metrics.
- Never refuses to start.

The contract — metric names exposed in users' dashboards stay pure
because no auto-generated fallback name leaks into the namespace.

## Probes mapped (all 15)

| Probe | OTel native used | senhub.* extensions |
|---|---|---|
| `cpu` | `system.cpu.time/utilization/load_*` (OTEP 0119) | DPC/IRQ/queue rates (Windows) |
| `memory` | `system.memory.{limit, usage, utilization}`, `system.paging.utilization` | paging rates, Windows-specific states |
| `network` | `system.network.{io, packet.count, errors, packet.dropped}` | none |
| `logicaldisk` | `system.filesystem.{limit, usage, utilization}` | inode metrics, Windows disk I/O rates |
| `ping_gateway`, `ping_webapp` | – | `senhub.probe.icmp.{duration_seconds, packet_loss_ratio}` |
| `load_webapp` | – | `senhub.probe.http.duration_seconds` (with `phase` label) |
| `wifi_signal_strength` | – | `senhub.system.network.wifi.{signal_strength.dbm, quality_ratio}` |
| `syslog`, `event`, `otel` | n/a — explicit `skip: true` (event conduits) | – |
| `redfish` | `hw.status`, `hw.physical_disk.size`, `hw.logical_disk.{limit, usage, utilization}` | RAID pools, redundancy, Windows-specific extensions, `hw_state="unknown"` extension |
| `veeam` | – | full `senhub.veeam.*` namespace (33 metrics → 20 OTel names via collapse) |
| `citrix` | – | full `senhub.citrix.*` namespace (45 metrics → 19 OTel names via collapse) |
| `netscaler` | `system.filesystem.*` (appliance disk) | full `senhub.netscaler.*` (100 metrics → ~65 OTel names; 11 status enums via expand) |

## Backward compatibility

- **PRTG / Nagios / SenHub strategies**: zero behavioral change. Same
  endpoints, same channel names, same JSON output, same probe types in
  configuration files (e.g., `type: logicaldisk` still works as before
  even though the metrics it now exposes also surface as
  `system.filesystem.*`).
- **JWT licenses**: every probe type ID is preserved (`logicaldisk`,
  `cpu`, etc.) — existing licenses keep working.
- **Configuration files**: enabling Prometheus is purely additive
  (`endpoints: [..., prometheus]` + optional `prometheus:` sub-block).
