# Release 0.1.96-beta

**Date:** 2026-05-19
**Type:** Beta Release (pprof endpoint + NDJSON log fix + prod smoke hardening)
**PRs merged to dev:** #125, #126, #127

## Overview

Three PRs merged since 0.1.95-beta. The release has two operational themes.

**Theme 1 — Visibility into the bbcloud post-restart stall (PRs #126, #127)**

The production stall on bbcloud (task #92) — all 5 probes go silent after the JT400 bridge
respawns, agent process stays alive but does nothing — had no available diagnosis path on
0.1.95-beta. This release ships two complementary fixes that together restore operator
visibility:

- PR #127 exposes pprof endpoints under the existing agentkey-authenticated prefix
  `/api/{agentkey}/debug/pprof/`. Operators can now capture goroutine dumps, CPU profiles, and
  heap snapshots from a live agent without restarting it or recompiling with custom flags. This
  is the prerequisite for investigating the stall: the root cause is almost certainly a blocked
  goroutine or a mutex held by a panicked or stalled goroutine after the JT400 respawn.

- PR #126 fixes the NDJSON line boundary issue in `MaskingWriter`. On bbcloud (Windows) the log
  file was a single 4 MB physical line containing ~27 000 concatenated JSON blobs because
  `MaskingWriter.Write` was stripping the trailing `\n` that zerolog sent. `tail -f`, `jq`,
  `Get-Content -Tail`, and all NDJSON log shippers (Loki, fluent-bit, Splunk forwarder) were
  either hanging or refusing the file entirely. The agent was effectively invisible to operations
  during the stall incident — the log file could not be read by any normal tool.

**Theme 2 — Production smoke test hardening (PR #125)**

The first real end-to-end run of `make prod-smoke` against the 0.1.95-beta deploy on sha901 +
bbcloud exposed five defects in the smoke harness itself. PR #125 fixes them all and records
the first green run: 3/3 tests pass, 133 s wall time. The harness is now reliable enough to use
as a regression gate after each beta deploy.

## Breaking changes

None. All changes are additive or internal. Existing configurations load and run unchanged.

## Features

### pprof endpoints under authenticated debug prefix (PR #127, commit `0b8366f`)

The HTTP strategy now mounts Go's standard profiling handlers under:

```
/api/{agentkey}/debug/pprof/
/api/{agentkey}/debug/pprof/cmdline
/api/{agentkey}/debug/pprof/profile
/api/{agentkey}/debug/pprof/symbol
/api/{agentkey}/debug/pprof/trace
/api/{agentkey}/debug/pprof/{name}   # goroutine, heap, allocs, threadcreate, block, mutex
```

Authentication uses the existing `AuthenticateAndExtract` helper — same security posture as the
rest of `/api/{agentkey}/debug/*`. No unauthenticated path is exposed.

Implementation deliberately avoids mounting `http.DefaultServeMux`: `net/http/pprof`'s `init()`
registers handlers on `DefaultServeMux` as a side effect, which would expose bare
`/debug/pprof/*` routes without auth. The named handlers are referenced directly instead.

Typical capture commands during the bbcloud stall investigation:

```bash
# Goroutine dump — identify which goroutines are blocked and where
curl "http://localhost:8080/api/$KEY/debug/pprof/goroutine?debug=2"

# 30-second CPU profile
curl -o cpu.pprof "http://localhost:8080/api/$KEY/debug/pprof/profile?seconds=30"

# Heap snapshot
curl -o heap.pprof "http://localhost:8080/api/$KEY/debug/pprof/heap"
```

No new dependencies (net/http/pprof is stdlib). No behavioural change when the endpoints are
not hit.

## Fixes

### NDJSON line boundaries dropped by MaskingWriter (PR #126, commit `29347a4`)

`zerolog` calls `Write` once per log entry; the payload is already terminated by `\n`.
`MaskingWriter.Write` was dropping that newline on the JSON fast-path: `json.Unmarshal`
silently ignores trailing whitespace; `json.Marshal` does not emit a newline. Successive entries
therefore concatenated into one giant physical line in the log file.

Observed on bbcloud: a 4 MB log "file" with ~27 000 entries on a single line.

Downstream consequences:

- `Get-Content -Tail N` on Windows scans backwards indefinitely looking for newlines that don't
  exist (it hung the prod_smoke tests on the first run)
- `tail -f`, `jq`, `less` all unusable
- NDJSON log shippers (Loki, fluent-bit, Splunk forwarder) reject the file as malformed

The fix adds `appendNewlineIfMissing(b)` — idempotent, called on the three write paths in
`MaskingWriter` (JSON happy path, JSON marshal error fallback, non-JSON raw path). Three new
tests pin the invariant: two NDJSON entries in → exactly two lines out; non-JSON input still
gets terminated; `appendNewlineIfMissing` is idempotent on already-terminated input.

Discovered while investigating the bbcloud post-jt400-respawn stall. The missing newlines meant
log aggregators did not realize the agent was alive, and on-disk grep/audit tooling was
underestimating incident severity.

### prod_smoke: Go RE2 has no negative lookahead (PR #125, commit `2b1dd0b`)

`secretKeyWithPlainValuePattern` used `(?!\*\*\*")` to exclude already-redacted values inline.
Go's regexp package is RE2, which deliberately omits negative lookahead — the regex panicked at
package init. Reshaped to a two-pass match: capture every `"<sensitive-key>":"<value>"` pair
with a plain RE2 expression, then filter in code (`value != "***"` → leak). Semantics are
identical; the package now compiles and the test runs correctly.

Discovered when running `make prod-smoke` for the first time post-Round-1 deploy.

## Internal

### prod_smoke: five hardening fixes from first end-to-end run (PR #125, commit `3c1ad0c`)

The first real run against the live 0.1.95-beta deploy on sha901 + bbcloud exposed five defects
in the harness:

**Concurrent buffer write (remoteShell)** — `Stdout=Stderr=&buf` is unsafe under `exec.Cmd`;
Go starts two writer goroutines, the buffer is not thread-safe, and on a 20 MB log payload the
IO pipeline deadlocked deterministically. Replaced with `CombinedOutput()`.

**No SSH timeouts (remoteShell)** — a stalled remote command (journalctl pager hang, sudo
prompt waiting on stdin) would block the harness indefinitely. Added
`ConnectTimeout=10 / ServerAliveInterval=15 / ServerAliveCountMax=4`; sessions now terminate
within ~60 s of silence.

**`Get-Content -Tail` on bbcloud hangs** — the Windows log file was a single physical line (see
NDJSON fix above), so `-Tail N` scanned backwards forever. Changed to `Get-Content -Raw` which
reads the full file as a single string and returns immediately.

**`sudo -n` for agent version check** — the agent's `main.go` privilege check refuses to run
without root. `sudo -n` (non-interactive) now makes the smoke test fail-fast if the NOPASSWD
policy regresses, instead of hanging on a password prompt.

**Post-restart scope for bearer + probe-params checks** — the agent does not rotate its log
file on restart; pre-deploy entries from a previous (vulnerable) agent version stay on disk.
`log_window.sinceLastRestart` locates the most recent `"Initializing agent in offline mode"`
marker and scopes the leak checks to the slice after it — what matters is what the currently
deployed agent emits.

**isAsterisks: two-layer mask acceptance** — `SanitizeParamsForLog` replaces sensitive values
with `"***"`, but the downstream `MaskSensitiveData` hook (pre-Round-1) then re-masks to
`"********"` (length-aware padding). The test now treats any all-asterisks value as masked
rather than asserting the exact `"***"` string.

   in #125 can be reverted to line-based `Get-Content -Tail` once 0.1.96-beta is deployed and

   credentials not in GitHub Secrets).

   growth rate. Plan: bump `compactVersion` to `0x02` and grow to `uint64` before the next probe
   batch lands.

   yet.
8. **Veeam "Job Details" chip missing status** — `category` split in 0.1.90 left the status
   field empty in the chip label. Fix deferred until next Veeam probe touch.
