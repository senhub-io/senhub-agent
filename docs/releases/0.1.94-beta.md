# Release 0.1.94-beta

**Date:** 2026-05-18
**Type:** Beta Release (OTLP backpressure hardening — Tier 1 + Tier 2 complete)
**PRs merged to dev:** #109, #110, #111, #112, #113, #114, #115

## Overview

Seven PRs merged since 0.1.93-beta, all centred on the OTLP strategy. The release closes out the
full OTLP backpressure roadmap (Tiers 1 and 2) that was opened after the sha901/pub400 stall
investigation. A second headline is the OTLP tooling layer (PR #114 + #115) that makes the new
counters operator-accessible directly from `agent status --otlp`, the web dashboard, and a new
`/api/{agentkey}/info/otlp` endpoint — without needing to scrape Prometheus.

**Theme 1 — OTLP backpressure (PRs #109–#113)**

The IBM i probe at sha901 exposed two failure modes: (a) the scheduler goroutine dying silently on a
panic left no trace in the log, and (b) a slow gRPC export let the LWW store grow unbounded until
memory pressure triggered the OS to kill the agent. Five PRs address these root causes in layers:

- **Tier 1** (PR #109): cardinality cap on the metric store, `store_size` / `export_duration` /
  `dropped` self-metrics, scheduler `recover()`, OTLP timeout default 10s → 60s.
- **Tier 2a** (PR #110): memory-limiter circuit breaker (soft/hard MiB thresholds), modelled on the
  OTel Collector `memory_limiter` processor.
- **Tier 2b** (PR #111): per-probe cardinality budget (`max_active_series_per_probe`, default 10 000)
  isolates a rogue probe from starving peers.
- **Tier 2c** (PR #112): persistent LWW checkpoint (atomic `.tmp` + `os.Rename`, configurable path
  and save interval). Dashboards see continuity across agent restarts, OOM kills, and OS reboots.
- **Tier 2d** (PR #113): split-batch parallel export by probe (`max_concurrent_exports`, default 4),
  bounded by a semaphore over the shared gRPC/HTTP2 connection.

**Theme 2 — OTLP tooling (PRs #114, #115)**

- New `agent status --otlp` CLI flag and `/api/{agentkey}/info/otlp` JSON endpoint expose the full
  OTLP pipeline snapshot without a Prometheus scrape.
- New "OTLP Pipeline" card on the web dashboard refreshes on the 30s cycle, with sub-tables for
  drop reasons and checkpoint error stages when non-empty.
- New `docs/admin-guide/OTLP-OBSERVABILITY.md`: field-level reference for every counter/gauge, every
  drop reason, every checkpoint error stage, starter alert recipes, and per-OS checkpoint paths.
- User-guide and CLI reference updated for the multi-file per-OS paths, `--otlp` flag, and the
  new monitoring section.

## Breaking changes

None. All new YAML knobs default to the pre-0.1.94-beta behaviour:

- `max_store_size: 50000` — unchanged from Tier 1 default already in 0.1.93-beta.
- `memory_limit` — disabled when both `soft_mib` and `hard_mib` are 0 (default).
- `max_active_series_per_probe: 0` — unbounded, same as before.
- `persistence.enabled: false` — opt-in; no files written by default.
- `max_concurrent_exports: 1` for `max_concurrent_exports <= 1` path matches the pre-2d serial
  behaviour. Default is 4 (parallel), but the serialised code path activates automatically when
  the record count is below the 100-record `SplitBatchThreshold`.

Existing agent configurations continue to load and run unchanged.

## Features

### OTLP Tier 1 — cardinality cap + backpressure visibility (PR #109)

- New `max_store_size` YAML knob (default 50 000; 0 = unbounded for back-compat). Once the LWW
  store reaches the cap, new series are dropped and the existing series continue to update.
- New self-metrics emitted on every push tick:
  - `senhub.agent.otlp.store_size` (gauge) — distinct series held at the most recent push.
  - `senhub.agent.otlp.export.duration{window="last"|"mean"}` (gauges) — wall-clock export time.
  - `senhub.agent.otlp.dropped{reason="store_cap|memory_soft_limit|memory_hard_limit|probe_cardinality"}`
    (counter) — drops before export, attributed by reason. Plug-in point for future reasons.
- Scheduler: `deferred recover()` so a panic in any probe's `Collect()` surfaces as an ERROR log
  (probe name + panic value) rather than silently killing the goroutine.
- OTLP default timeout bumped from 10s to 60s — confirmed fix for the sha901/pub400 stall
  (IBM i's ~2000-datapoint cycle consistently timed out under the 10s budget).

### OTLP Tier 2a — memory limiter circuit breaker (PR #110)

- Background goroutine polls `runtime.MemStats.HeapAlloc` every `check_interval` (default 5s).
- Soft state (`HeapAlloc >= soft_mib`): refuse new series; existing series continue to update.
- Hard state (`HeapAlloc >= hard_mib`): refuse all writes + force a GC pass to try to recover.
- Lock-free atomic state read on the upsert hot path.

```yaml
otlp:
  memory_limit:
    soft_mib: 200        # 0 disables; default 200
    hard_mib: 400        # 0 disables; default 400
    check_interval: 5s
```

### OTLP Tier 2b — per-probe cardinality budget (PR #111)

- `max_active_series_per_probe` (default 10 000; 0 = unbounded) caps the number of distinct series
  any single probe instance can hold in the shared store.
- Probe counts tracked via a `probe_name` attribute; records without `probe_name` (agent
  self-metrics) are excluded from per-probe budgeting.
- Upsert ordering: memory pressure (atomic) → existence check → probe budget → global cap → insert.
  The cheapest guard wins; existing series are never re-checked.

```yaml
otlp:
  max_store_size: 50000
  max_active_series_per_probe: 10000
```

### OTLP Tier 2c — persistent LWW checkpoint (PR #112)

- Opt-in on-disk snapshot of the metric store. Saves every `interval` (default 30s); restores at
  boot so dashboards see continuity across restarts, OOM kills, and OS reboots.
- Atomic write via `.tmp` + `os.Rename` — a crash mid-write leaves no half-written file.
- Versioned JSON format: a reader seeing a higher version logs a warn, starts with an empty store.
- Restore bypasses upsert guards (the snapshot is trusted state); probe counts are rebuilt to match.
- Four new self-metrics on every tick: `checkpoint.size`, `checkpoint.last_save_age`,
  `checkpoint.restored_entries`, `checkpoint.errors{stage=read|parse|...}`.

```yaml
otlp:
  persistence:
    enabled: true
    path: /var/lib/senhub-agent/otlp
    interval: 30s
```

### OTLP Tier 2d — split-batch parallel export (PR #113)

- Optional fan-out of one push cycle into per-probe sub-batches, each exported on its own goroutine
  over the shared gRPC connection (HTTP/2 multiplexes — no per-goroutine connection pool needed).
- `max_concurrent_exports: 4` default. Validated range: [1, 64].
- Single-batch path activates automatically when `max_concurrent_exports <= 1` or record count is
  below `SplitBatchThreshold` (100) — avoids goroutine overhead for small payloads.
- Agent self-metrics (no `probe_name` attribute) ride their own sub-batch (`__agent__` partition).
- Partial-success (some partitions fail) still propagates an error and bumps the export-errors
  counter.
- New self-metric: `senhub.agent.otlp.parallel.sub_batches` (gauge) — number of sub-batches the
  last push fanned out across. 1 = single-batch path; >1 = parallel path fired.

```yaml
otlp:
  max_concurrent_exports: 4
```

### OTLP tooling — CLI + web UI + HTTP endpoint (PR #114)

- `agent status --otlp` prints a 4-section table after the standard status output: pipeline
  counters, store + export-duration, checkpoint state, parallel export stats.
- New `/api/{agentkey}/info/otlp` JSON endpoint returns the same snapshot (auth-gated by agent
  key).
- New "OTLP Pipeline" card on the web dashboard: refreshes on the 30s cycle; sub-tables for
  drops-by-reason and checkpoint-errors-by-stage appear only when non-empty.

### Documentation (PR #115)

- `docs/admin-guide/OTLP-OBSERVABILITY.md` — field-level reference: every counter/gauge, every
  drop reason, every checkpoint error stage, starter alert recipes, per-OS checkpoint paths.
- Multi-file config page: top-of-page callout + per-OS path table (Linux systemd, Linux tarball,
  Windows, macOS). Clarifies that `.d/` directories resolve next to `agent.yaml`.
- CLI Reference: new "Status" sub-section documenting `--otlp` output and interpretation hints.
- User-guide OTLP section: new "Monitoring the OTLP pipeline" sub-section with a
  symptom → field troubleshooting table and pointers to the cardinality/memory knobs.

## Fixes

- **OTLP export timeout** (PR #109, commit `c584b4b`): default 10s → 60s. Root cause of the
  sha901/pub400 stall: IBM i's ~2000-datapoint export systematically exceeded the budget; the
  metric store kept growing each cycle without ever successfully flushing. Confirmed fixed with
  13+ minutes of uninterrupted data flow post-change.
- **Scheduler silent death** (PR #109, commit `c584b4b`): a panic in `Collect()` previously killed
  the scheduler goroutine with no log entry. Now surfaces as an ERROR and the goroutine continues.
- **Web dashboard OTLP card placement** (PR #114, commit `8b192b8`): card was wired to the
  wrong template and never rendered. Fixed to target the actually-served dashboard template.
- **CLI `--otlp` flag parsing** (PR #115, commit `34ba980`): flag was parsed via a walk of
  `os.Args` that bypassed the canonical arg parser. Threaded through `cliArgs.ParsedArgs` /
  `ShowOTLP bool` so it survives any normalisation pass.
- **Import grouping** (PR #115, commit `34ba980`): three files touched by PR #114 had third-party
  and internal imports in the same group. Corrected.

## Internal

- **`tools/license-mint`** (commit `1e48296`): dev-only `go run` tool to generate compact-format
  licenses for testing without a round-trip to the Sensor Factory mint endpoint. Not compiled into
  the agent binary.
- **`sort.Strings` over hand-rolled insertion sort** (commit `34ba980`): `sortedKeys` in the
  CLI formatter replaced with the stdlib call.
- Self-metric count for the OTLP strategy is now 24 (up from the tier-1 count). Agent-metrics
  test expected-count updated accordingly.

## Known follow-ups

1. **Tier 3 documentation** — the roadmap item for Prometheus-pull as the recommended architecture
   for fleets >>50 probes has not been written yet. This remains a 0.2.x item.
2. **Cumulative counter continuity across restart** — `start_time` resets on agent restart; the
   persistent checkpoint preserves LWW values but not the OTel SDK's cumulative counter epoch.
   Consumer-side scrape gaps are still visible. Orthogonal to this work; tracked separately.
3. **Compact license bitmap width** — at the current growth rate (~5 probes per release) bit 31
   will be claimed within ~3 releases. Plan: bump `compactVersion` byte from `0x01` to `0x02` and
   grow to `uint64` before the next batch of probes lands.
4. **IBM i real-customer smoke test** — sha901/pub400 is a public demo system. A test against an
   actual customer IBM i (Heliaq or similar) is still outstanding for end-to-end validation.
5. **Zabbix output** — Phase 1 starting next release window. Not advertised in user-facing docs yet.
6. **Veeam "Job Details" chip missing status** — `category` split in 0.1.90 left the status field
   empty in the chip label. Fix deferred until next Veeam probe touch.
