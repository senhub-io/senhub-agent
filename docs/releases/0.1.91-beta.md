# Release 0.1.91-beta

**Date:** 2026-05-13
**Type:** Beta Release (Prometheus output cleanup + UI fixes)
**Branch merged to dev:** `fix/prometheus-output`

## Overview

This release fixes **11 bugs** in the Prometheus output format
uncovered during runtime testing of 0.1.90 across macOS, Linux and
Windows. The fixes ship as four logical commits:

1. **Pipeline hygiene** — network throughput / packet / error metrics
   shipped as cumulative counters but were actually emitted as
   per-second rates, breaking PromQL `rate()` and `increase()`.
   They now expose as gauges with `/s` units. Memory `state="free"`
   no longer appears twice in the scrape output. The `unit` cache
   tag no longer leaks as a Prometheus label.
2. **Naming + cardinality + filtering** — `system.linux.cpu.load_*`
   renamed to `system.cpu.load_*` so macOS hosts no longer emit a
   "linux" metric. Secondary IP addresses no longer create
   positional `ip_1, ip_2, … ip_N` labels. macOS APFS firmlink
   volumes (`/System/Volumes/iSCPreboot`, `xarts`, `Hardware`,
   `Update`) are filtered from filesystem metrics.
3. **Web UI** — the API Explorer's *Output Format* dropdown now
   correctly routes Prometheus responses through the text-output
   path (was JSON-parsed → garbled error) and constructs the
   probe-less URL `/api/{key}/prometheus/metrics` (was
   `/api/{key}/prometheus/metrics/{probe}` → 404).
4. **CPU semantics across OS** — `cpu_user`, `cpu_system`,
   `cpu_idle`, `cpu_nice`, `cpu_iowait`, `cpu_irq`, `cpu_softirq`,
   `cpu_steal` used to emit cumulative seconds on Unix and
   percentages on Windows, yet share the same YAML mapping
   (`system.cpu.time` counter, unit `s`). On Windows the resulting
   `_total` counter fluctuated 0-100 and broke every monotonicity
   contract. The probe + YAML now agree on **instantaneous per-mode
   utilization** (`system.cpu.utilization` gauge with the
   `cpu.mode` attribute, unit `1`).

The dead Windows perfmon raw-name entries in `cpu.yaml`
(`processor_time`, `user_time`, `privileged_time`,
`interrupt_time`, `dpc_time` and four "Legacy per-core" copies)
are removed — the probe renames those PDH paths before pushing to
the cache, so they could never resolve at runtime.

## Breaking changes

### Prometheus metric names

| Before (0.1.90) | After (0.1.91) | Reason |
|---|---|---|
| `senhub_system_network_io_bytes_total` | `senhub_system_network_io_bytes_per_second` | gauge, not counter |
| `senhub_system_network_packet_count_total` | `senhub_system_network_packet_count_per_second` | gauge, not counter |
| `senhub_system_network_errors_total` | `senhub_system_network_errors_per_second` | gauge, not counter |
| `senhub_system_network_packet_dropped_total` | `senhub_system_network_packet_dropped_per_second` | gauge, not counter |
| `senhub_system_linux_cpu_load_1m` (+5m, 15m) | `senhub_system_cpu_load_1m` (+5m, 15m) | not Linux-only |
| `senhub_system_cpu_time_seconds_total{cpu_mode="user"}` | `senhub_system_cpu_utilization_ratio{cpu_mode="user"}` (and 7 other modes) | gauge ratio, not counter seconds |

### Probe behaviour

- **Unix cpu probe** now diffs `cpu.Times()` between two collect
  cycles to compute per-mode percentages. The very first
  collection emits no per-mode datapoints — they appear from the
  second tick. Total CPU utilization (`cpu_usage_total`) and
  per-core metrics still emit on tick 1 (they come from gopsutil's
  blocking `cpu.Percent` API).
- **Unix memory probe** no longer emits `memory_available` — the
  metric was conflicting with `memory_free` in the OTel pipeline.
  PRTG/Nagios users running on Linux who had a "Memory Available"
  channel mapped to it will need to migrate to `memory_free` (same
  semantic — `gopsutil` reports identical values for both on Linux
  unless backed by /proc/meminfo's `MemAvailable`).
- **Network probes (Unix + Windows)** drop secondary IPs from tags.
  Interfaces are now identified by their first valid (non-link-local)
  IP only. If you were relying on `ip_1, ip_2, …` labels in a
  Grafana / PRTG view, switch to the primary `ip` label.
- **logicaldisk probe (macOS only)** skips firmlink/synthetic
  volumes under `/System/Volumes/` (except `/System/Volumes/Data`).
  This is darwin-specific; Linux behaviour unchanged.

### Web UI

- **API Explorer** — selecting *Prometheus* in the Output Format
  dropdown no longer requires a probe selection; the URL becomes
  the agent-wide `/api/{key}/prometheus/metrics`.

## Migration guide

### Grafana / PromQL queries

Search-and-replace in your dashboards / alert rules:

```promql
# Network throughput
- senhub_system_network_io_bytes_total
+ senhub_system_network_io_bytes_per_second

# CPU per-mode (no more rate() needed — already instantaneous)
- rate(senhub_system_cpu_time_seconds_total{cpu_mode="user"}[1m])
+ senhub_system_cpu_utilization_ratio{cpu_mode="user"}

# Load average (drop the linux. infix)
- senhub_system_linux_cpu_load_1m
+ senhub_system_cpu_load_1m
```

Network rate metrics are now expressed as `bytes_per_second` etc.
Drop any wrapping `rate()` — the agent does the differentiation.

### PRTG / Nagios users

PRTG and Nagios output paths are **unchanged** for network, memory
(non-`memory_available`), load average, and APFS filtering. The
only PRTG/Nagios-visible change is on Unix CPU mode channels:
`CPU User`, `CPU System`, `CPU Idle`, `CPU Nice`, `CPU IO Wait`,
`CPU Irq`, `CPU Softirq`, `CPU Steal` switch from cumulative
seconds to instantaneous percentages (0-100). This aligns Unix
output with what Windows has reported since day one and matches
what humans expect when reading "CPU User".

## Runtime validation

The release was validated end-to-end on three operating systems:

| OS | Host | Method | Result |
|---|---|---|---|
| macOS 25 (darwin) | local mac mini | direct scrape via `/api/{key}/prometheus/metrics` | 7 of 8 testable bugs OK; bugs #10/#11 not observable (gopsutil `cpu.Times` not implemented on darwin) |
| Linux (Ubuntu) | sha901 | direct scrape | 11/11 bugs validated, 121 series, 0 duplicates |
| Windows Server 2022 | bbcloud | direct scrape | 11/11 bugs validated, 77 series, 0 duplicates, all 4 cpu modes + per-core utilization between 0-1 |

Math sanity check on Linux: per-mode utilization sums to ~1.0
(idle 77 %, user 21 %, system 1.4 %, plus tiny iowait/softirq/steal
fractions), confirming the delta computation is correct.

## Test additions

- `TestUnixCollector_CPUTimes_PercentageMath` — pins the delta /
  totalDelta × 100 formula so a regression to cumulative seconds
  fails immediately.
- `TestUnixCollector_CPUTimes_FirstCallNoEmit` — confirms the
  first collect cycle seeds `lastTimes` without emitting bogus
  zero or 100 % datapoints.
- `TestSerialize_FiltersInternalUnitAttribute` — pins the `unit`
  filter so 0.1.91 does not regress.

Total new tests: **3**. Full test suite green (37 packages, 0
failures). Cross-platform builds verified: darwin / linux /
windows.
