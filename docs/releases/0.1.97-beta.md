# Release 0.1.97-beta

**Date:** 2026-05-19
**Type:** Beta Release (ibmi probe restricted to Linux)

## Overview

Single-change release: the `ibmi` probe is now **Linux-only** on the agent side. Windows and macOS builds of the agent still ship, but the `ibmi` probe constructor refuses to instantiate on those platforms. Operators running mixed-OS fleets must move the IBM i workload to a Linux host.

## Why this change

During the 0.1.96-beta production validation on the bbcloud Windows host, we captured the following pattern from production logs:

- 5 `jt400 bridge subprocess died; respawn on next query` events in a 3-hour window — every respawn cleanly recovered.
- 9 `probe.ibmi` cycles with `doCall` durations > 30s. **Three of them measured at 2252s, 2051s, and 2604s** (37–43 minutes per cycle).
- The agent's log file contained **two 37-minute gaps with no log activity at all** — not just from the ibmi probe, but from every probe — that lined up exactly with the start and end of those slow cycles.
- The agent process stayed alive (CPU 0 %, normal RSS), the Windows OS had no sleep state available (`powercfg /a` reports no S1/S2/S3/S0/hibernate states; `powercfg /lastwake` shows 0 wake events), the W32Time service logged no clock jumps, and the host had 85 days of uninterrupted uptime.

The agent was therefore genuinely silent for ~37 minutes at a time while the Go process kept running. Root cause is still unclear; pprof endpoints (shipped in 0.1.96-beta via PR #127) are now wired into bbcloud but the live stall capture has not yet caught a recurrence.

Rather than continue to ship a probe whose only known production exposure on Windows produces operator-invisible 37-minute log silences, we pull Windows support for the `ibmi` probe outright. The Linux path continues to be exercised daily on `sha901` (`pub400.com` monitor, 84 metric series flowing through OTLP → otelcol → VictoriaMetrics → Grafana with no observed stalls).

## What changes

| Surface | Before | After |
|---|---|---|
| `NewIBMiProbe` on Windows / macOS | Spawned the JT400 bridge subprocess | Returns `ibmi probe is only supported on linux (current platform: windows); remove type: ibmi from your probes config or run the agent on a Linux host` |
| Release assets | `jt400runner-windows-amd64.exe` shipped | Asset dropped from the GitHub Release |
| Native runner workflow matrix | `linux-amd64`, `linux-arm64`, `windows-amd64` | `linux-amd64`, `linux-arm64` only |
| `senhub-agent` binary for Windows / macOS | Still shipped | Still shipped — every other probe is unaffected |

## Operator impact

If you currently run the agent on Windows or macOS **without** an `ibmi` probe in `probes.yaml` / `probes.d/`, **no action required** — your deployment is unchanged.

If you currently run the agent on Windows or macOS **with** an `ibmi` probe configured:

1. The agent will start and log a clear error explaining the probe is unsupported on the current OS.
2. Every other probe in the same config will run normally.
3. To re-enable IBM i monitoring, deploy a small Linux host (a VM with 1 vCPU / 1 GB RAM is sufficient) running the `senhub-agent` and configure the `ibmi` probe there. The IBM i partition itself does not need to be Linux — the probe connects via JT400/JDBC to the IBM i system regardless of where the agent runs.

## Internal / structural

- `internal/agent/probes/ibmi/platform_linux.go` and `platform_other.go` introduce a build-tag-gated `platformGate()` helper. The `linux_logs` probe uses the same pattern (`internal/agent/probes/linuxlogs/journal_reader_other.go`).
- `TestNewIBMiProbe_RefusesOnNonLinux` and `TestPlatformGate_LinuxAllows` together pin the contract on both sides.
- `master-release.yml`, `dev-beta-release.yml`, and `native-runner-build.yml` drop their `windows-amd64` native runner references.
- `docs/admin-guide/IBMI-RUNTIME-BUILD.md` documents the supported matrix at the top.

## Known follow-ups

- Root cause of the bbcloud stall remains uninvestigated. The pprof endpoint is still armed on `sha901` and `bbcloud`; if a stall recurs on a Linux host (none observed to date), capture a goroutine dump via `curl http://127.0.0.1:18080/api/$KEY/debug/pprof/goroutine?debug=2` and diff against the baseline at `/tmp/bbcloud-baseline-goroutines.txt`.
- If a real demand for Windows / macOS `ibmi` support surfaces, re-enable per platform after each gets its own stall-resilience test pass (long-running fuzz of bridge respawn under simulated slow stdin/stdout).
