# Release 0.2.0-beta

**Date:** 2026-05-19
**Type:** Major version — breaking changes
**PRs merged to dev:** TBD (single PR bundling Phase A + B + C)

## Overview

Major housekeeping release. Three breaking-change axes bundled in a single PR so the migration is a single step rather than three sequential ones:

1. **Offline-only mode** — the SaaS-configuration code path is gone. The agent only loads its configuration from local YAML files.
2. **OS-canonical paths** — config + log directories align with each OS's filesystem conventions.
3. **ZIP-only release artifacts** — the GitHub release ships per-platform ZIPs, not raw binaries.

## Breaking changes

### CLI flags removed

The pre-0.2.0 flags below now produce a hard parse error at startup:

| Flag | Replacement |
|---|---|
| `--offline` | Removed — offline is the only mode |
| `--authentication-key` | Removed — the agent key lives in the config file, auto-generated at install |
| `--server-url` | Removed — the cloud intake URL is build-time injected |

**Operators MUST update their `ExecStart` (systemd) or service path (Windows SCM) before upgrading.** Go's `go-arg` parser rejects unknown flags and the service won't start otherwise.

Migration of an existing systemd unit:

```diff
- ExecStart=/usr/local/bin/senhub-agent run --offline --config-path /etc/senhub-agent/agent.yaml
+ ExecStart=/usr/local/bin/senhub-agent run --config-path /etc/senhub-agent/agent.yaml
```

Migration of the Windows service binPath:

```diff
- "C:\SenHub\senhub-agent.exe" run --offline
+ "C:\SenHub\senhub-agent.exe" run
```

### Default config path (Linux)

Pre-0.2.0 the default config path was `<binary_dir>/agent-config.yaml` — typically `/usr/local/bin/agent-config.yaml`, which violates FHS. 0.2.0+ uses:

| OS | Default `--config-path` |
|---|---|
| Linux | `/etc/senhub-agent/agent.yaml` |
| Windows | `%ProgramData%\SenHub\agent.yaml` |
| macOS | `/usr/local/etc/senhub-agent/agent.yaml` |

Existing installs that pass `--config-path` explicitly are unaffected. Existing installs that relied on the legacy default need to either move the file to the new canonical location or keep passing `--config-path` explicitly.

### Log directory (Linux only)

Pre-0.2.0 path: `/var/log/senhub/`. 0.2.0+ path: `/var/log/senhub-agent/`.

Existing installs continue to work because the log directory is lazy-created on first write; the lumberjack rotator will simply start writing to the new path. The cleanup-on-uninstall logic was extended to scan both old and new paths so an upgrade-then-uninstall removes the legacy directory too.

### Release artifact layout

Pre-0.2.0 each release shipped:

```
senhub-agent_linux_amd64               (raw binary)
senhub-agent_linux_amd64.zip           (containing the raw binary)
… × 5 platforms = 10 agent artifacts
```

0.2.0+ ships only the ZIPs, and the binary inside is always named `senhub-agent` / `senhub-agent.exe` (no platform suffix):

```
senhub-agent-linux-amd64.zip           (containing: senhub-agent)
senhub-agent-linux-arm64.zip           (containing: senhub-agent)
senhub-agent-windows-amd64.zip         (containing: senhub-agent.exe)
senhub-agent-darwin-amd64.zip          (containing: senhub-agent)
senhub-agent-darwin-arm64.zip          (containing: senhub-agent)
```

Any external tooling that downloads by URL needs to switch from the pre-0.2.0 URL pattern (`/download/<version>/senhub-agent_<os>_<arch>`) to the new one (`/download/<version>/senhub-agent-<os>-<arch>.zip`) and add a ZIP extraction step.

### Auto-updater path

The auto-updater (`agent update <version>` and the in-service auto-update timer) now downloads the ZIP, extracts the binary, and applies the upgrade via the same `selfupdate.Apply` atomic-rename pattern. **A 0.2.0+ agent cannot auto-update from a pre-0.2.0 release** — the old URL pattern points at a file that no longer exists. Operators upgrading from 0.1.x must do one manual replace; from 0.2.0+ onwards auto-update resumes normally.

## Install-time directory creation

`agent install` now eager-creates the following directories so the first `agent run` finds a fully-formed layout:

- `<config-dir>/probes.d/` (empty, ready for fragments)
- `<config-dir>/strategies.d/` (idem)
- Log directory (`/var/log/senhub-agent`, `%ProgramData%\SenHub\logs`, `/Library/Logs/SenHub`)

Pre-0.2.0 the log directory was created lazily on first write; the `.d/` directories were never created and operators had to `mkdir` manually before dropping fragments.

## Removed internal symbols

For library consumers (none expected externally, internal only):

- `configuration.RemoteConfiguration` and the entire SaaS-loader machinery (641 lines deleted).
- `configuration.NewRemoteConfiguration`, `NewMockRemoteConfiguration` (replacement: `NewMockConfiguration` with the same shape).
- `configuration.NewAgentConfigurationWithRemote`.
- `agent.DetectAgentMode`, `agent.LocalConfigInfo`.
- `AgentConfiguration.GetServerUrl()` — the cloud intake URL is build-injected as `cliArgs.ProductionURL` / `DevelopmentURL` (helper: `cliArgs.CloudIntakeURL()`).

The shared data type names (`RemoteConfigurationData`, `StorageConfig`, `ProbeConfig`, `AgentConfig`) are kept as-is to limit the rename surface in this release. A follow-up will rename them once the consumers (data store, sensor pool, http strategy, …) are migrated.

## Test coverage changes

- Removed: `remoteConfiguration_test.go` + `remoteConfiguration_replication_test.go`.
- Added: `paths_<os>.go` helpers exposing `LogBaseDir()` and `canonicalConfigPathForOS()`; tested via the existing cliArgs test suite.
- Updated: auto-update tests now expect the ZIP URL pattern.

`make test` is green. `go vet ./...` clean. Cross-compile clean on linux/amd64, linux/arm64, windows/amd64, darwin/amd64, darwin/arm64.

## Migration runbook for the operator

1. Stop the senhub-agent service.
2. Update the service unit (systemd) or Windows service binPath to drop `--offline` (and `--authentication-key` / `--server-url` if present).
3. If the install relied on the pre-0.2.0 default config path (`<binary_dir>/agent-config.yaml`), either:
   - move the file to `/etc/senhub-agent/agent.yaml` (Linux) / `%ProgramData%\SenHub\agent.yaml` (Windows); or
   - keep the explicit `--config-path` argument.
4. Download `senhub-agent-<os>-<arch>.zip` from the 0.2.0-beta release.
5. Unzip → replace the binary.
6. Start the service.
7. Verify with `senhub-agent version` and `tail` of the log file.

## Known follow-ups

- The `RemoteConfigurationData` / `RemoteConfiguration*` type names should be renamed away from the "Remote" prefix; tracked for a 0.2.1 cleanup PR.
- The auto-updater's pre-0.2.0 raw-binary path was hard-removed; if a customer needs in-place auto-upgrade from 0.1.x to 0.2.x, the workaround is a one-time manual binary replace.
