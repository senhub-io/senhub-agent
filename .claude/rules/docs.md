## Three audiences, three trees

| Path | Audience | Tone | Examples |
|---|---|---|---|
| `docs/user-guide/` | End user of an agent / SenHub customer | Task-oriented, screenshots, copy-pastable YAML | Configuration page, probe pages, web UI overview |
| `docs/admin-guide/` | Operator deploying & running agents | Operations, troubleshooting, HTTPS / logging / backups | HTTP-STRATEGY, LOGGING, HTTPS-CONFIGURATION |
| `docs/developer-guide/` | Internal — engineers writing probes / strategies | Design rationale, semconv, test strategy | architecture, design-patterns, otel/senhub-semantic-conventions |

Don't blur the lines. A user-guide page that drops into Go struct names is in the wrong tree.

## Canonical references

When in doubt about a name / shape / convention, these are the authoritative sources:

- **OTel naming** → `docs/developer-guide/otel/senhub-semantic-conventions.md` (684 lines, updated per lot). All probe metric names must trace back to this doc.
- **Database probes catalog** → `docs/developer-guide/database-probes/METRICS.md` + `DESIGN.md`.
- **Time-series key design** → `docs/developer-guide/engineering/TIME_SERIES_KEY_DESIGN.md` + `DISCRIMINANT-TAGS-REGISTRY.md`.
- **Config versioning** → `docs/admin-guide/CONFIG-VERSION-CHANGELOG.md`.

When you change a contract documented in one of these files, update the doc in the same commit. Stale developer-guide content is worse than no content — readers calibrate trust on it.

| Icinga | **Don't advertise**. Compatible via Nagios output but not officially tested. |

## Release notes

- One file per release: `docs/releases/X.Y.Z-beta.md`.
- Structure: Overview, Breaking Changes (with migration), Features, Fixes, Internal/Refactor, Known Follow-ups.
- For metric-name changes, ALWAYS include a before/after table.

## Hugo build artifacts

`docs/user-guide/public/` contains pre-rendered HTML and search indices. They're regenerated by `hugo build`. **Don't hand-edit anything under `public/`** — it'll be overwritten on the next build. If you spot stale content there, it means the next deploy needs a rebuild.

## Documentation index

`docs/DOCUMENTATION-INDEX.md` is the cross-tree map. When you add a major doc, add a one-line link there so the new file is discoverable from the index.

## Style notes

- French is fine in internal developer notes (memory, design docs, release notes). User-facing docs are **English only** — the SenHub customer base is international.
- Code examples in user-guide should be runnable / copy-pastable. Use `yaml`, `bash`, `go` code fences with the language tag.
- Don't include `→ in metric names` or `⚠` symbols in user-guide tables — they don't render consistently across the Hugo theme + RSS feed + PDF export.
