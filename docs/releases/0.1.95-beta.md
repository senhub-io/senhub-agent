# Release 0.1.95-beta

**Date:** 2026-05-18
**Type:** Beta Release (Round 1 security — log redaction + downgrade prevention + IBM i runtime resolver)
**PRs merged to dev:** #116, #117, #118, #119, #120, #121, #122, #123

## Overview

Eight PRs merged since 0.1.94-beta. The release has two themes:

**Theme 1 — Round 1 security hardening (PRs #120–#123)**

Three security fixes triggered by incidents observed on bbcloud and sha901 during the Phase B
deployment of 0.1.94-beta:

- The OTLP strategy was logging the full gRPC error string on auth failures; because the OTel
  collector echoes the Bearer token back in the error message, each failed push tick leaked the
  token into the agent log file. Fixed by a regex redactor on the cold error path (PR #120).
- The probe-startup log line serialised the full `params` map, leaking user identifiers and
  credential-named keys verbatim. A sanitiser now masks any key matching the secret-name pattern
  before the value reaches zerolog (PR #121).
- The auto-update timer could silently downgrade a running agent when the registry's `latest`
  alias pointed to an older release — the exact incident on bbcloud where 0.1.94-beta was
  replaced by 0.1.91. The update gate now enforces strict-greater-than semver and fails closed
  on parse errors (PR #122).

PR #123 adds a `prod_smoke` integration suite that asserts all three fixes hold on live agents
post-deploy, without requiring CI access to production credentials.

**Theme 2 — IBM i runtime resolver + native runner build (PRs #116–#119)**

- PR #116 fixed Windows CI failures in the IBM i bridge and config tests that were blocking PR
  checks since the IBM i probe landed.
- PR #117 expanded CI trigger rules to run the Go test workflows on pull requests to `dev`
  (they were previously only running on push).
- PR #118 replaces the IBM i bridge's hard-coded `jt400runner` path with a resolver that
  probes a priority list of platform-appropriate locations and surfaces operator-friendly errors
  on failure. A new `agent ibmi check` CLI subcommand runs the resolver and reports the result.
- PR #119 adds a GraalVM native-image build workflow that produces a self-contained
  `jt400runner` binary for Linux (eliminates the JVM prerequisite on the monitoring host) and
  documents the build process in `docs/admin-guide/IBMI-RUNTIME-BUILD.md`.

## Breaking changes

None. All changes are additive or internal. Existing configurations load and run unchanged.

## Features

### IBM i — resolver-based runtime detection (PR #118)

The IBM i bridge now locates the `jt400runner` binary through a priority-ordered resolver
rather than expecting a fixed path. The resolver checks platform-appropriate candidate locations
(system PATH, well-known install directories, probe config `runtime_path` override) and returns
the first valid executable found.

On failure the resolver emits a structured error listing every candidate tried, so operators
know exactly which paths were checked and can correct their installation without reading source.

New CLI subcommand:

```bash
agent ibmi check
```

Runs the resolver, prints the resolved path (or the structured failure), and exits. No agent
startup required — useful for installation validation and troubleshooting.

### IBM i — GraalVM native-image build workflow (PR #119)

New CI workflow (`.github/workflows/native-runner-build.yml`) builds `jt400runner` into a
GraalVM native image for Linux amd64. The resulting binary is self-contained — no JVM or
JDK required on the monitoring host.

Build instructions and download notes are in `docs/admin-guide/IBMI-RUNTIME-BUILD.md`.

## Fixes

### OTLP Bearer token leak in export-failure log (PR #120, commit `6736fda`)

When the OTel collector rejects an OTLP push with an auth error, the gRPC error message echoes
the offending token: `scheme or token does not match: Bearer <token>`. The strategy was logging
that string verbatim via zerolog's `Err()` and forwarding it to the span status message,
surfacing the bearer in every log aggregator pointed at the agent.

A small regex redactor (`Bearer` + non-whitespace → `Bearer ***`) is now applied on the cold
error path before the structured log field and the span status. The regex does not touch log
lines that contain the word "bearer" without a following token, so legitimate prose in error
messages survives untouched.

Discovered during Phase B on bbcloud: a 30-second auth mismatch produced a full token leak per
push tick into a log file routinely tailed through shared infrastructure.

### Probe params leak in 'Starting new probe' log (PR #121, commit `4d4e367`)

The sensor's probe-startup log line serialised the full `probe.Params` map via `zerolog.Any()`,
which echoed every value verbatim. User identifiers (e.g. the IBM i `user:` field), API tokens,
login names, and credential strings reached the log file as plaintext.

`configuration.SanitizeParamsForLog(map)` produces a shallow copy of the params map and masks
any key whose name matches `(?i)(key|token|password|secret|user|login|email|credential)`. The
runtime config map is never mutated; only the log-bound view is sanitised. Applied at the
`sensor.go` call site.

Discovered on bbcloud where the IBM i probe's `user:` value leaked on every probe start.

### Auto-update downgrade prevention (PR #122, commit `014ee69`)

**This is the P0 fix for this release.** The `Update()` method previously triggered on any
version-string mismatch, regardless of direction. When the registry's `latest` alias resolved
to a release older than the running agent, the agent silently replaced its own binary.

Reproduced on bbcloud during Phase B: a stale `agent-config.yaml` from a legacy online-mode
install caused the auto-update timer to fire with online-mode defaults. An agent on 0.1.94-beta
was silently downgraded to 0.1.91 mid-deployment.

The fix introduces `shouldUpdateTo(current, expected) (bool, error)`:

- Strict-greater-than semver comparison via `hashicorp/go-version` (already an indirect
  dependency). Pre-release ordering is respected: `0.1.91 < 0.1.94-beta < 0.1.94`.
- Fails closed on parse errors: an unparseable version string logs a Warn and returns without
  triggering the update. The previous behaviour — proceed on unparseable input — was the
  proximate cause of the production incident.
- Two explicit log messages distinguish "skipped because not newer" from "skipped because
  unparseable", so operator logs are unambiguous.

### Windows CI — IBM i bridge and config tests (PR #116, commit `326e60e`)

The IBM i bridge tests and the `config show` test were failing on the Windows CI runner due to
path-separator assumptions and a missing `filepath.ToSlash` normalisation. Fixed; Windows
checks are now consistently green.

## Internal

### Production smoke-test suite (PR #123, commit `5c3e29c`)

New `tools/prod_smoke/` package, gated by the `prod_smoke` build tag, provides three
post-deploy integration tests that assert the Round 1 fixes hold on live agents:

- `TestBearerLeak_NoBearerInLog` — scans the last 5000 log lines on each host for bearer
  token patterns; asserts none survive.
- `TestProbeParamsLeak_NoUserInLog` — locates every "Starting new probe" entry and asserts
  that no sensitive key name carries a non-`***` value.
- `TestNoDowngrade_AgentIsAtExpectedVersion` — reads `senhub-agent version` on each host and
  asserts it matches `SENHUB_EXPECTED_AGENT_VERSION` (default `0.1.95-beta`).

Tests shell out over SSH via `sshpass`; credentials are resolved through
`~/.senhub/read-secret.sh` — no embedded secrets. A test SKIPs (not FAILs) when credentials
for a given host are absent, so the suite stays runnable from a workstation without full
production access.

Run with: `make prod-smoke`. Not wired to CI — deliberately operator-triggered.

### CI trigger expansion (PR #117)

The `go-test.yml` and `go-test-windows.yml` workflows now also run on pull requests targeting
`dev`, in addition to the existing push triggers. PR checks are now consistently enforced
before merge.
